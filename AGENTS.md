# AGENTS

## 项目定位

本仓库用于商用项目交付，默认目标是可上线、可运维、可审计、可扩展。
所有计划、技术方案、代码生成与实施决策，必须以商用落地为第一优先级。

## 最高优先级原则

1. 稳定性优先：避免引入高不确定性的技术方案。
2. 安全与合规优先：默认要求鉴权、审计、数据保护、风控与合规提示。
3. 可维护性优先：接口命名、字段语义、模块边界、错误码体系保持一致。
4. 可观测性优先：关键链路必须具备日志、指标、追踪与告警设计。
5. 可回滚优先：高风险操作必须设计幂等、防重、补偿或回滚预案。
6. 成本可控优先：在满足商用品质前提下控制云资源、第三方服务与人力成本。

## 明确禁止

1. 禁止以“学习价值”替代商用稳定性与交付确定性。
2. 禁止输出仅概念化文档而缺少可执行接口和字段定义。
3. 禁止在未说明风险的情况下引入新基础设施依赖。
4. 禁止跳过安全边界设计（权限、速率限制、输入校验、审计记录）。

## 文档与工程生成要求

1. 接口文档必须满足后端可直接生成 Controller、DTO、Service 骨架。
2. 字段必须明确类型、单位、必填性、枚举范围、校验规则。
3. 金额字段统一使用分为存储与传输单位，前端展示再转元。
4. 时间字段统一使用 ISO 8601（UTC）字符串。
5. 每个高风险操作必须有错误码、失败语义与用户提示建议。

## 多端协同要求

覆盖端：web-admin、web-institution、h5-staff、mp-student。

1. 优先抽象四端共享领域接口，减少重复与分叉。
2. 端侧专属接口必须在文档中显式标注调用方。
3. 跨端联动流程必须有状态一致性定义（如结算、禁用、审核）。

## 基础设施约束

1. 当前环境基线：腾讯云轻量服务器 + 域名 happymaa.cn + COS static.happymaa.cn。
2. 后续将接入短信验证码服务，设计时需预留 provider 抽象与降级策略。
3. 两个 web 端、一个 h5 端、一个小程序端将共享同一套后端与部署体系。

## 变更控制

1. 任何关键决策变更需在文档中记录原因、影响范围与回滚方案。
2. 涉及接口破坏性变更必须采用版本化策略并提供迁移窗口。
