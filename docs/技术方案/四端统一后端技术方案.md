# 四端统一后端技术方案（商用版）

## 1. 文档目标

本方案用于指导后端服务按统一契约生成工程，覆盖以下客户端：

- web-admin（平台运营后台）
- web-institution（机构管理门户）
- h5-staff（业务员工作台）
- mp-student（学员小程序）

目标：

1. 提供可执行技术选型与模块边界。
2. 提供可直接生成后端骨架的 REST 接口与字段类型。
3. 明确状态机、错误码、审计与非功能约束，满足商用上线要求。

## 2. 技术选型

### 2.1 后端与基础设施

- Runtime: Node.js 22 LTS
- Framework: NestJS 11
- ORM: Prisma 6
- DB: MySQL 8.0
- Cache: Redis 7
- Auth: JWT（AccessToken）+ Refresh Token
- Validation: class-validator + class-transformer
- API 文档: Swagger(OpenAPI 3.1) 自动生成（源定义为 TypeScript DTO）
- 日志: pino（结构化 JSON）
- 追踪: OpenTelemetry（HTTP + DB + Redis）
- 定时任务: BullMQ / cron（结算、账务状态巡检）

### 2.2 前端约束（统一建议）

基础栈已确定：React + Ant Design。
建议补充库：

- Router: react-router-dom
- 请求层: axios（统一拦截器）
- 服务状态管理: @tanstack/react-query
- 全局轻状态: zustand
- 图表: echarts + echarts-for-react
- 日期: dayjs
- 表单增强: react-hook-form（可选，与 antd Form 二选一）

## 4. 统一接口规范

### 4.1 API 版本

- Base URL: /api/v1
- 破坏性变更必须升版本。

### 4.2 鉴权模型

- Header: Authorization: Bearer <token>
- Role 枚举：PLATFORM_ADMIN、INSTITUTION_ADMIN、STAFF、STUDENT
- 高风险接口需 RBAC + 操作审计。

### 4.3 请求/响应包装

```ts
export interface ApiSuccess<T> {
  code: 0;
  message: "OK";
  requestId: string;
  timestamp: string; // ISO 8601 UTC
  data: T;
}

export interface ApiError {
  code: number; // 业务错误码
  message: string;
  requestId: string;
  timestamp: string;
  details?: Record<string, unknown>;
}

export interface PageQuery {
  page: number; // >=1
  pageSize: number; // 1~100
}

export interface PageResult<T> {
  items: T[];
  total: number;
  page: number;
  pageSize: number;
}
```

### 4.4 字段规则

1. 金额字段：单位为分，类型 bigint（接口传 number，超过安全范围时传 string）。
2. 比率字段：百分数，最多两位小数（0~100）。
3. 时间字段：ISO 8601 UTC 字符串。
4. 状态字段：必须用枚举。
5. 主键：使用 string（推荐雪花 ID 或 ULID）。

## 8. 状态机约束

1. Institution: PENDING -> ACTIVE -> SUSPENDED -> EXITED（不允许反向恢复，恢复需新审批流程）。
2. Salesman: ACTIVE -> DISABLED（当前版本不可恢复）。
3. Settlement: PENDING -> SETTLED（不可撤销）。
4. Order: CREATED -> COOLING_OFF -> ACTIVE -> COMPLETED，任意阶段可进入 TERMINATED/REFUNDED。
5. Period: PENDING -> DELIVERED -> PAID，DELIVERED 可超时转 OVERDUE，OVERDUE 可核销为 WRITTEN_OFF。

## 10. 审计、安全与可观测

1. 审计日志

- 高风险接口必须记录：operatorId、role、action、resourceId、before、after、reason、requestId。

2. 安全要求

- 登录与短信接口按 IP + phone 双维限流。
- 敏感字段加密存储（如身份证号脱敏存储）。
- 所有写接口启用参数校验与 XSS/SQL 注入防护。

3. 可观测

- 每个请求注入 requestId。
- 指标至少包含：QPS、P95、错误率、第三方依赖成功率。
- 对短信、签约、人脸服务设置失败告警阈值。
