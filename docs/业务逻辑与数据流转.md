# 先学后付·网课超市 — 业务逻辑与数据流转

> 本文描述五端协作的完整业务逻辑、数据状态机与跨系统数据流向。  
> 合规基线：卖方/合同甲方/收款方/开票方均为**机构**，平台不垫资、不归集资金。

## 二、用户认证与身份核验

### 2.1 登录方式

| 端 | 登录方式 | 接口 | Token 角色 |
|----|---------|------|-----------|
| mp-student | 微信/支付宝小程序 code | `POST /v1/auth/wechat/login` `POST /v1/auth/alipay/login` | `student` |
| mp-staff | 微信/支付宝小程序 code | 同上（`appType: "staff"`） | `staff` |
| web-institution | 手机号 + 密码 | `POST /v1/auth/org/login` | `org` |
| web-admin | 用户名 + 密码 | `POST /v1/auth/admin/login` | `admin / ops / finance` |

**微信登录数据流：**
```
小程序 wx.login()
  → code
  → POST /v1/auth/wechat/login { code, appType, referrerStaffId?, referrerStudentId? }
  → 后端调微信 /jscode2session 换取 openid
  → 按 openid 查 student 或 staff 表
  → 若学员不存在：自动建档，写入 referrerStaffId / referrerStudentId
  → 若业务员不存在：拒绝登录（必须预录入）
  → 签发 JWT { sub, role, exp: 7d }
  → 返回 { token, student/staff, isNew }
```

### 2.2 实名认证（学员）

**触发时机：** 学员首次下单前必须完成。  
**合规约束：** 年龄 < 18 岁硬拒绝，不得绑过。

```
POST /v1/users/realname { name, idNo, phone }
  → 四要素验证（姓名 + 身份证 + 手机 + 银行卡，调第三方实名 API）
  → 解析身份证中的出生年月，计算年龄
  → age < 18 → 抛 ForbiddenException（未成年拦截，不可绕过）
  → 更新 student:
      realnameVerified = true
      ageVerifiedAdult  = true
      idNoHash          = SHA256(idNo)        // 用于唯一性校验
      idNoEncrypted     = AES(idNo, APP_KEY)  // 用于实名展示
```

### 2.3 JWT 鉴权机制

所有需要身份的接口均使用 `JwtAuthGuard`：
- `Authorization: Bearer <token>` → 解码 payload → 注入 `request.user`
- `@CurrentUser('id')` 装饰器从 `request.user` 取字段
- 数据隔离在 Service 层强制过滤（如机构只能查自己的订单：`where: { sellerOrgId: orgId }`）

## 三、主线业务流程

### 3.1 学员购课完整链路

```
[学员进入小程序]
  ↓ gate 页：必须携带 referrerStaffId（来源拦截）
  ↓ 微信/支付宝一键登录 → 自动建档（写入 referrerStaffId）
  ↓
[实名认证]（仅首次）
  POST /v1/users/realname
  student.realnameVerified = true, ageVerifiedAdult = true
  ↓
[浏览课程]
  GET /v1/courses?keyword=&page=&size=        ← 公开接口，无需鉴权
  GET /v1/courses/:id
  ↓
[创建订单]
  POST /v1/orders { courseId }
  前置检查：
    ① student.realnameVerified && ageVerifiedAdult
    ② course.status === 'ONLINE'
    ③ course.price ≤ platform_config.price_cap
    ④ 业务员自买防护：referrerStaff.phone === student.phone → attributionStaffId = null
  写入 order:
    status = CREATED
    sellerOrgId = course.orgId      ← 卖方强绑机构
    periodAmount = ceil(price / periodCount)
  ↓
[信用授权]（分期场景）
  POST /v1/credit/authorize { orderId, scenario: "INSTALLMENT_COURSE" }
  → 调芝麻/微信支付分 API 查询额度与风险等级
  → 写入 credit_authorization + credit_decision
  ↓
[签署合同]
  POST /v1/orders/:id/sign
  前置检查：order.status === CREATED
  操作：
    ① 创建 contract { partyAOrgId=机构, partyBStudentId=学员, esignProvider=esign }
    ② order.status → COOLING_OFF
    ③ order.coolingOffDeadline = now + 7天
  ↓
[冷静期 7 天]
  期间学员可 POST /v1/orders/:id/refund → 无条件退款
    order.status → REFUNDED
    级联删除 PENDING/HELD 状态的 commission + referralReward
  冷静期结束（定时任务每日凌晨扫描）：
    order.status → ACTIVE
    调 installmentService.generateInstallments()：
      生成 periodCount 条 InstallmentItem
      dueDate = now + n×30天
      status = PENDING
  ↓
[分期学习阶段]（见第四、五、六节）
  第1期内容自动解锁 → 学员打卡 → 代扣 → 解锁下期内容 → 循环
  ↓
[全部期次 PAID → order.status = COMPLETED]
```

### 3.2 订单状态机

```
                    ┌────────────────────────────────────┐
                    │            Order Status             │
                    └────────────────────────────────────┘

 createOrder()          signContract()        冷静期到期(定时任务)
CREATED ──────────────► COOLING_OFF ─────────────────────► ACTIVE
                            │                                  │
                  refund()  │                    所有期次PAID  │
                            ▼                                  ▼
                        REFUNDED                          COMPLETED

 任意阶段异常终止
ACTIVE / COOLING_OFF ──► TERMINATED
```

| 状态 | 含义 | 允许操作 |
|------|------|---------|
| CREATED | 已下单，未签约 | 签合同、取消 |
| COOLING_OFF | 冷静期中 | 无条件退款（7天内） |
| ACTIVE | 分期履约中 | 打卡、代扣、逾期处理 |
| COMPLETED | 全额结清 | 查看记录 |
| REFUNDED | 冷静期退款 | 只读 |
| TERMINATED | 中途终止 | 只读 |

## 四、分期计划与代扣

### 4.1 分期计划生成

```
order.status 从 COOLING_OFF → ACTIVE 时触发

generateInstallments(orderId):
  for n in 1..periodCount:
    InstallmentItem {
      orderId
      periodNo      = n
      dueDate       = now + n×30天
      planAmount    = order.periodAmount
      actualAmount  = planAmount - incentiveApplied（默认0）
      status        = PENDING
      contentDeliveredAt = null   ← 代扣前置条件，必须先置位
    }
```

### 4.2 代扣触发条件（合规核心）

> **先交付后扣款** — 代扣指令必须在 `contentDeliveredAt` 已置位后才可发起。  
> 违反此条件返回 error code **42201**。

```
triggerDeduction(installmentItemId):
  item = findInstallmentItem(id)
  if !item.contentDeliveredAt:
    throw 42201 "内容未交付，不允许发起代扣"   ← 合规强卡点
  执行代扣（见4.3）
```

### 4.3 多渠道代扣与重试

**渠道优先级：** 微信支付分 → 芝麻先享 → 银行卡代扣

```
executeDeduction(item):
  channels = [WECHAT_SCORE, ZHIMA, BANKCARD]
  for channel in channels:
    try:
      调对应 SDK 发起扣款
      payee = item.order.sellerOrgId   ← 收款方恒为机构
      if success:
        item.status   → PAID
        item.deductedAmount = item.actualAmount
        触发 commissionService.settleOnPayment()   ← 业绩提成结算
        if item.periodNo === 1:
          触发 referralService.triggerReward()     ← 拉新奖励
        return
    catch:
      记录尝试日志（deduction.attempts JSON）
  // 所有渠道失败
  item.status → OVERDUE
  通知机构进行人工催收
```

**定时重试策略：**

| 定时任务 | 触发时间 | 行为 |
|---------|---------|------|
| `runDailyDeductions` | 每日 09:00 | 扫描当日到期 & contentDeliveredAt 已置位的 PENDING 项，首次代扣 |
| `retryOverdueDeductions` | 每日 10:00 | 逾期项按 D+1 / D+3 / D+7 三个窗口重试，满4次失败则定格 OVERDUE |

### 4.4 分期项状态机

```
PENDING
  │ markContentDelivered()
  ▼
DELIVERED（contentDeliveredAt 已置位）
  │ 代扣成功
  ▼
PAID
  │
  └── 代扣失败（重试耗尽）→ OVERDUE → 机构处理 → WRITTEN_OFF（核销坏账）
```

## 五、内容交付与学习打卡

### 5.1 内容解锁（代扣前置）

```
POST /v1/learning/:orderId/unlock { periodNo }
  验证 studentId 与 orderId 归属
  installmentItem.contentDeliveredAt = now   ← 置位，允许代扣
  写 ServiceRecord { type: CONTENT_UNLOCK }
  返回解锁的视频/资料链接
```

**触发逻辑：**
- 第1期：订单进入 ACTIVE 时自动解锁
- 后续期：上一期代扣成功（PAID）后自动解锁下一期

### 5.2 扫脸打卡

```
POST /v1/learning/checkin { orderId, coursePeriod, faceToken }
  验证 studentId 归属
  faceMatch(student.idNoEncrypted, faceToken):
    生产：调腾讯云/e签宝人脸核身 API
    开发：直接通过
  写 ServiceRecord { type: CHECKIN }
  发放学习激励：
    LearnIncentive { amount: 500分(5元), source: sellerOrgId }
    累计激励可在下期还款时抵扣（incentiveApplied）
```

### 5.3 数据流：内容交付 → 打卡 → 代扣 → 下期解锁

```
[机构/系统] markContentDelivered(itemId)
                  ↓
     installmentItem.contentDeliveredAt = now
                  ↓
[学员] 小程序展示课程内容（视频/资料）
                  ↓
[学员] 完课 → 扫脸打卡 → POST /v1/learning/checkin
                  ↓
     ServiceRecord(CHECKIN) + 发放学习激励
                  ↓
[定时任务/机构触发] POST /v1/internal/deduction/run
                  ↓
     前置检查 contentDeliveredAt ✓
                  ↓
     多渠道扣款（收款方=机构）
                  ↓
     PAID → 解锁下期内容 → 循环
```

## 六、业务员提成体系

### 6.1 提成类型

| 类型 | 枚举值 | 触发时机 | 金额 | 状态流转 |
|------|--------|---------|------|---------|
| 成单提成 | `CLOSING` | 签约（`signContract`）时创建 | 订单总额 × 3% | PENDING → 首期还款后 SETTLED |
| 业绩提成 | `PERFORMANCE` | 每期代扣成功时创建 | 当期金额 × 1% | PENDING → 当期结算后 SETTLED |

### 6.2 状态流转

```
PENDING（提成已生成，待结算）
  │ 每期还款成功
  ▼
SETTLED（已结算）    ◄── 若订单冷静期退款，直接 deleteMany(PENDING/HELD)

PENDING / SETTLED
  │ 订单逾期（holdOnOverdue）
  ▼
HELD（暂缓）
  │ 后续还款恢复
  ▼
SETTLED

SETTLED
  │ 连续逾期 >60天（clawBackIfExtendedOverdue）
  ▼
CLAWED_BACK（扣回）
```

### 6.3 自购防护

```
createOrder():
  if referrerStaff.phone === student.phone:
    attributionStaffId = null   ← 订单不关联业务员，不产生提成
```

## 七、拉新奖励

### 7.1 奖励规则

- **单级奖励**（非多级，反传销）
- **触发条件：** 被邀请学员首次成功还款（`periodNo === 1` 且代扣 SUCCESS）
- **奖励金额：** 50 元毛额（5000 分）→ 代扣 20% 个税 → 40 元净入账
- **防刷：** 同一被邀请人 + 订单只触发一次

### 7.2 数据流

```
[学员A] 生成邀请链接
  POST /v1/referral/invite
  → 返回 /register?inv={studentA.id}
                     ↓
[学员B] 扫码进入小程序
  POST /v1/auth/wechat/login { referrerStudentId: studentA.id }
  → student_B.referrerStudentId = studentA.id
                     ↓
[学员B 首期还款成功]
  deductionService.executeDeduction(item, periodNo=1)
  → referralService.triggerReward(inviteeId=B, orderId)
                     ↓
  校验：
    ① referralReward 不存在重复记录
    ② inviter(A) 手机号不在 Staff 表（业务员不走此通道）
  写入 ReferralReward {
    inviterStudentId: A,
    inviteeStudentId: B,
    orderId,
    grossAmount:  5000,
    taxWithheld:  1000,   // 20%
    netAmount:    4000,
    status:       PENDING,
    trigger:      "invitee_first_repayment"
  }
```

## 八、机构端数据流

### 8.1 机构准入

```
[机构] 提交资质材料（营业执照、教育资质）
  ↓ org.status = PENDING
[平台运营] POST /v1/admin/orgs/:id/approve { feeRate: 0.03 }
  ↓ org.status = ACTIVE, org.settlementFeeRate = 0.03
[机构] 可登录 web-institution，创建课程
```

### 8.2 课程生命周期

```
POST /v1/courses/org { title, price, periodCount, outline, ... }
  → course.status = OFFLINE（默认下线）
PUT /v1/courses/org/:id { status: 'ONLINE' }
  → 学员可见，可下单
PUT /v1/courses/org/:id { status: 'OFFLINE' }
  → 已购订单继续履约，新订单不可创建
```

### 8.3 逾期处理

```
[机构] GET /v1/org/orders?status=OVERDUE（筛选逾期订单）
[机构] POST /v1/org/overdue/:installmentItemId/action { action: "write_off", remark: "..." }
  → installmentItem.status = WRITTEN_OFF
  → 写 ServiceRecord { type: TEACHER_OUTREACH }
  → 机构承担坏账损失（平台不兜底）
```

### 8.4 月度结算

```
[定时任务] 每月1日生成 Settlement { orgId, period, gmv, platformServiceFee, status: PENDING }
  gmv             = 当月所有 PAID 分期项之和
  platformServiceFee = gmv × org.settlementFeeRate
  应付机构        = gmv - platformServiceFee

[平台财务] POST /v1/admin/settlement/:id/settle
  → settlement.status = SETTLED
  → 平台向机构转账应付金额（对接银行转账 API）
```

## 九、平台保证金机制

```
机构入驻时预存保证金（depositBalance）
  ↓
当出现学员投诉/强制退款/严重逾期时：
  DepositLedger { type: RISK_DEDUCTION, amount: -xxx, reason: "..." }
  org.depositBalance -= xxx
  ↓
保证金不足时，平台可 suspend 机构
  POST /v1/admin/orgs/:id/suspend
  → org.status = SUSPENDED
  → 机构无法新增订单，学员存量订单继续履约
```

## 十一、关键合规约束汇总

| 约束 | 实现位置 | 说明 |
|------|---------|------|
| 卖方=机构 | `createOrder` | `sellerOrgId = course.orgId`，不可覆盖 |
| 收款方=机构 | `executeDeduction` | `payee = order.sellerOrgId`，不可覆盖 |
| 先交付后扣款 | `triggerDeduction` | `!contentDeliveredAt → throw 42201` |
| 未成年拦截 | `verifyRealname` | `age < 18 → ForbiddenException`，硬编码 |
| 7天冷静期 | `signContract` / `coolingOffRefund` | 签约后7天内无条件退款 |
| 客单价上限 | `createOrder` | `course.price > config.price_cap → 拒绝` |
| 业务员自购防护 | `createOrder` + `createClosingCommission` | 同手机号置空 attributionStaffId |
| 单级分佣 | `referralService` | 无递归/多级逻辑 |
| 业务员重复拦截 | `triggerReward` | inviter 手机号在 Staff 表 → 不发拉新奖励 |
| 信用分不入库 | `credit.service` | 只存授权状态，不存原始分值 |
| 冷静期退款级联 | `coolingOffRefund` | 删除所有 PENDING/HELD 提成与拉新奖励 |
