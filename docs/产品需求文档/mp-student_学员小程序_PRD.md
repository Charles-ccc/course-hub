# mp-student · 学员小程序 PRD

> **版本：v2.1** | 更新日期：2026-06-29  
> 本文档按三个迭代阶段组织，Iteration 1 为 MVP，Iteration 2 为核心功能完善，Iteration 3 为高级合规与体验升级。

---

# 1. 背景

学员小程序是「欢乐学」面向学员的核心消费端，目前仅支持支付宝小程序。
前端技术实现采用支付宝原生小程序方案，不使用跨端框架。

学员通过业务员的邀请链接或二维码进入小程序，完成注册、浏览课程、免预付报名、阶段履约和在线学习的完整消费旅程。「先学后付」是本产品最核心的商业模式：学员先学习本阶段课程，阶段结束后系统自动代扣学费，降低学员的决策门槛。实名认证与电子签约作为独立合规能力逐步补齐。

## 1.1 功能范围

- 归因追踪：在应用启动时捕获来源（业务员/学员邀请），写入归因数据
- 注册与认证：支付宝授权注册（手机号免填）、登录页支付宝授权登录、个人中心实名认证入口与状态展示
- 课程发现：关键词搜索、课程列表、课程详情和免费试学
- 下单报名：创建订单、芝麻先享授权
- 学习：视频课播放（Iter2）、联系老师、人脸打卡获取学习激励（Iter3）
- 订单管理：查看订单列表、扣款计划进度
- 个人中心：查看个人信息、实名认证入口、退出登录

## 1.2 价值

对学员：先体验后付款，降低购课决策风险；电子签约保障权益透明。  
对平台：归因追踪保障业务员提成准确计算；实名认证、电子签约与履约记录构建合规基础。

# 4. Iteration 1 — MVP 需求

## 4.0 需求背景与业务目标

### 4.0.1 背景

为保障用户来源可控，避免无效注册，平台采用**业务员邀请制注册**模式。用户必须通过**机构码 / 业务员邀请码**完成注册，注册后支持多种登录方式。所有注册用户必须归属某一业务员或机构，作为后续业绩统计、提成计算的数据基础。

### 4.0.2 业务目标

- 所有注册用户必须绑定业务员，确保用户来源可控
- 防止非目标用户随意注册，降低无效流量
- 记录完整的邀请关系链，支撑业绩追踪与提成计算
- 降低登录门槛，支持多种登录方式，提升留存率
- 提升注册转化率，缩短注册链路

### 4.0.3 用户角色

| 角色       | 说明                                                 |
| ---------- | ---------------------------------------------------- |
| 普通学员   | 被邀请注册的小程序用户，通过业务员邀请码完成注册     |
| 业务员     | 拥有机构码和邀请码，负责邀请学员，后续与学员提成挂钩 |
| 后台管理员 | 管理业务员、机构码状态、邀请关系审核                 |

### 4.0.4 业务流程（文字版）

**【未注册学员】**

```
打开小程序 → [自动捕获 staffId]
→ 本地无 token → 跳登录页
→ 用户点击「支付宝授权登录」→ my.getAuthCode 获取 authCode
→ 后端查 openId → 未注册
→ 展示注册引导页
→ 用户输入机构码 → 后端校验机构码
→ 展示机构信息 + 业务员名称
→ 用户点击「授权手机号」（支付宝原生授权组件）
→ 后端解密获得手机号 → 创建账号 → 绑定业务员
→ 跳首页（个人中心可补做实名认证）
```

**【已注册学员】**

```
打开小程序 → 检测本地 token
→ 有 token 直接跳首页；无 token 展示登录页
→ 用户点击「支付宝授权登录」→ POST /auth/alipay/login { authCode }
→ 后端查 openId：已注册 → 返回 token → 跳首页
→ token 过期时自动刷新，无需用户操作
```

## 4.1 应用启动与归因

作为平台，我希望在学员首次进入小程序时自动捕获来源归因，以便准确记录学员与业务员的绑定关系和学员互邀关系。

验收条件：

1. 应用启动时（onLaunch），解析启动参数中的 `?staff=<staffId>` 字段，若存在则写入本地存储键 `referrerStaffId`。
2. 启动参数中含 `?inv=<studentId>` 时，写入本地存储键 `referrerStudentId`。
3. 归因数据在后续注册请求时一并传入，其中 `referrerStaffId` 透传为 `inviteSalesmanId`；后端仅在首次注册成功时写入绑定关系，已绑定关系不可被后续请求覆盖。
4. 启动时检测本地是否有 accessToken：
   - 有 → 直接跳首页 Tab；后续 401 时跳转登录页
   - 无 → 跳转登录页 `/pages/auth/login/index`
5. 登录页用户主动点击「支付宝授权登录」按钮，触发 `my.getAuthCode` → `POST /api/v1/auth/alipay/login { authCode }`：
   - 已注册 → 保存 token，跳首页 Tab
   - 未注册 → 跳引导页 → 注册页

## 4.2 注册功能

### 4.2.1 机构码校验

作为学员，我希望通过输入机构码完成身份验证，以便确保我被正确的机构和业务员邀请。

验收条件：

**校验流程**

1. 注册表单首项为「机构码」输入框，用户输入机构码后，实时请求 `POST /api/v1/auth/org-code/validate { orgCode }`。
2. 校验项包括：机构码是否存在、是否启用（status = ACTIVE）、是否过期（expireAt > now）。
3. 校验成功后，页面展示确认卡片：「您将加入：XX 机构（业务员：张三）」，展示机构名称和关联业务员信息。
4. 校验失败时，根据错误码展示对应提示（见 4.7 异常处理）。

**限流与防暴力**

5. 同一支付宝 openId 在 24 小时内机构码校验失败 ≥ 5 次，冻结该账号机构码输入框 30 分钟；冻结期间提示「尝试次数过多，请 30 分钟后重试」。
6. 冻结记录存储至后端，端上不得绕过限流。

---

### 4.2.2 支付宝手机号授权注册

作为学员，我希望通过支付宝一键授权手机号完成注册，无需手动输入手机号或验证码。

验收条件：

**注册流程**

1. 机构码校验通过后，页面展示「授权手机号并注册」按钮（使用支付宝原生 `<button open-type="getAuthorize" scope="phoneNumber">` 组件）。
2. 用户点击按钮后，支付宝弹出授权确认框；授权成功后前端获得加密的手机号数据（`encryptedData` + `iv`），传给后端解密。
3. 用户拒绝授权时，提示：**需要授权手机号才能完成注册**，按钮保持可点击状态。
4. 后端解密手机号后，校验该手机号是否已注册：
   - 已注册（同一 openId）→ 直接返回 token，完成登录
   - 已注册（不同 openId）→ 返回错误码 40004，提示：**该手机号已被其他账号绑定，请联系客服**
   - 未注册 → 创建新账号

**提交与结果**

5. 提交 `POST /api/v1/auth/alipay/register { authCode, encryptedData, iv, orgCode, inviteSalesmanId }`，其中 `inviteSalesmanId` 来自本地存储的 `referrerStaffId`。
6. 注册成功后，后端自动写入邀请关系（student → inviteSalesmanId）；已绑定关系不可被后续请求覆盖。
7. 注册成功后将 token 和 profile 写入本地存储，跳转至首页；未实名用户后续可从个人中心进入实名认证页（`/pages/auth/realname`）。
8. 注册失败时页面展示接口返回的错误信息。

## 4.3 登录功能

> 登录完全依托支付宝身份体系，无需密码、无需短信，用户无感知完成登录。

### 4.3.1 支付宝授权登录页

验收条件：

1. 启动小程序时，splash 检测本地是否有 accessToken。无 token 跳登录页 `/pages/auth/login/index`；有 token 直接进首页 Tab（401 时再回登录页）。
2. 登录页 UI：logo + 品牌名 + 单一「支付宝授权登录」按钮 + 用户协议链接。
3. 点击按钮调用 `my.getAuthCode({ scopes: 'auth_base' })` 拿 authCode，请求 `POST /api/v1/auth/alipay/login { authCode }`：
   - 已注册用户 → 后端返回 `accessToken` + `refreshToken` + `profile`，保存 token 后跳首页 Tab
   - 未注册用户 → 后端返回 `needRegister: true`，跳转注册引导页
4. 用户拒绝授权时提示「需要支付宝授权才能登录」，按钮可点击重试。
5. Access token 过期时，自动用 refresh token 换新 token，无感重试；refresh token 也过期则清空本地 token 并跳回登录页。

## 4.4 找回密码

> 本产品不使用密码体系，登录完全通过支付宝身份凭证完成，**找回密码功能不适用，本节已废除**。
> 
> 若学员更换支付宝账号等特殊情况，需通过客服人工处理。

## 4.5 邀请关系管理与数据记录

验收条件：

1. 每个注册用户绑定的业务员信息存储至后端数据库，字段：`userId`、`phone`、`inviteSalesmanId`、`inviteSalesmanName`、`registeredAt`、`sourceType`（STAFF_CODE / STUDENT_INVITE）。
2. 邀请关系在注册时一次性确定，后续不可修改。
3. 若注册时未提供 `inviteSalesmanId`，注册仍允许通过，该用户标记为「无来源」。
4. 已绑定的邀请关系不可被后续请求覆盖。

## 4.6 业务规则与限制

### 4.6.1 注册限制

| 规则                     | 说明                                           |
| ------------------------ | ---------------------------------------------- |
| 无机构码不可进入注册流程 | 注册首页强制机构码校验，校验失败不显示后续字段 |
| 禁止同一手机号重复注册   | 若手机号已注册，拦截并引导登录                 |
| 一个机构码可邀请无限用户 | 可在后台配置单个机构码的邀请上限（可选）       |
| 邀请关系在注册时确定     | 已绑定关系不可被后续请求覆盖                   |

### 4.6.2 登录限制

| 规则             | 说明                                         |
| ---------------- | -------------------------------------------- |
| 唯一登录方式     | 支付宝 authCode 授权登录，用户在登录页主动触发 |
| 无密码体系       | 不存在密码、重置密码等概念                   |
| Token 有效期     | access token 15 分钟，refresh token 7 天     |
| Refresh 过期处理 | 重新调用 my.login() 获取新 authCode 完成续期 |

## 4.8 实名认证（独立认证能力，当前不阻塞报名）

> 当前阶段不再依赖支付宝开放认证，注册和报名流程也不因未实名而中断。实名认证作为独立合规能力保留入口与状态展示，后续复用 `/users/realname/initialize`、`/users/realname/confirm`、`/users/realname/callback` 接入正式服务。

验收条件：

1. 注册成功后直接进入首页，不强制跳转实名认证页。
2. 个人中心根据实名状态展示不同 UI：
   - 未实名：显示「完成实名认证」入口
   - 认证驳回：显示「实名认证未通过，重新认证」入口
   - 已实名：姓名旁展示「✓ 已实名」徽标
3. 点击实名认证入口跳转至 `/pages/auth/realname`，页面展示说明文案：**实名认证用于完善账户合规信息，当前不影响报名与学习**，并提供「开始认证」按钮。
4. 当前阶段若后端实名接口仍返回 501 或占位状态，前端提示：**实名认证功能即将上线**，用户可返回个人中心继续使用其他功能。
5. 后续正式接入独立实名服务时，继续复用现有 realname initialize / confirm / callback 接口；认证通过后将 `realnameStatus` 置为 `VERIFIED`，回写脱敏姓名等必要结果；认证失败或用户取消时允许重试。
6. 如后续实名结果判定未满 18 周岁，后端返回错误码 40001（UNDERAGE_USER），限制继续报名分阶段课程，并展示明确提示。

## 4.9 课程发现

### 4.9.1 课程列表（首页）

验收条件：

1. 首页顶部提供搜索框，支持输入关键词后点击搜索或键盘确认触发搜索。
2. 搜索或首次加载时请求 `GET /api/v1/courses?keyword=<关键词>&page=1&pageSize=20`，响应统一 `{ items, page, pageSize, total, hasNext }`。
3. 课程卡片展示字段：封面（imageUrl 缺省时显示占位图）、课程名称、机构名称、每期价格（元/期，计算：priceCents / periodCount / 100）、课程总价（元）、「先学后付」徽标。
4. 列表为空时展示空状态提示。
5. 进入小程序前必须登录（未登录由 splash 跳转登录页），首页不存在「未登录浏览」状态。
6. 点击课程卡片跳转至课程详情页。

### 4.9.2 课程详情

验收条件：

**页面内容**

1. 展示字段：课程名称、机构名称、课程总价（元）、每阶段金额（元）、阶段数、课程大纲（文本）、合规声明（平台为居间方，机构为合同甲方）。
2. 页面底部提供「免费试学」和「免预付报名」按钮。
3. 所有课程统一采用先学后付（DEFERRED）付款方式，不提供立即付款选项。

**免费试学（Iter 1 占位）**

4. 点击「免费试学第一节」，跳转至学习页，传入参数 `trial=1`；Iter 1 阶段学习页展示章节列表骨架与占位提示「视频功能即将上线」；Iter 2 补全实际播放。

**购课确认**

### 4.10.1 下单确认

> **决策（2026-06-25）：**
> - 允许同一学员对同一课程重复下单（后台管理员处理异常订单）。

## 4.15 MVP 验收标准

- 无机构码无法进入注册流程
- 机构码校验失败时展示明确错误提示
- 注册成功即绑定业务员，邀请关系不可覆盖
- 同一手机号禁止重复注册（同一 openId 只能注册一次）
- 已注册用户打开小程序，凭本地 token 直接进首页；token 失效跳登录页，点一次「授权登录」即可恢复
- 支付宝授权手机号一键完成注册，无需填写验证码或密码
- 所有异常均有明确的用户提示文案
- 登录、注册链路响应时间 ≤ 500ms
- 个人中心展示实名状态与认证入口，未实名不阻塞报名
- 课程列表与详情可正常访问
- 订单创建流程可走通（统一先学后付）
- 订单列表与详情可正常查看
- 学习页展示章节列表骨架，章节点击提示「视频功能即将上线」
- 订单详情页展示芝麻先享授权入口，授权完成后订单变为 ACTIVE
- 个人中心展示用户信息、实名状态，退出登录正常
