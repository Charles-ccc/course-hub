## 1. 文档目的与范围

本文档面向研发、测试与运维团队，描述"先学后付·网课超市"平台从注册到分期履约全链路的工程实现。范围包括四端（学员端、机构端、业务员端、平台后台）、后端服务、数据模型、对外集成（支付/代扣、电子合同、实名、视频点播、信用服务）、安全合规与非功能要求。

不在本文档范围：机构自身的课程生产、机构与学员之间分期债权的会计处理（由机构承担与核算）、最终法律合规结论（由律师出具）。

## 2. 合规设计基线（开发必读）

下列约束是系统级不变量，需在代码、数据库约束、配置与流程校验中强制落实：

1. **交易主体**：订单的卖方、合同甲方、发票开具方、收款方均为机构。`order.seller_org_id` 必填；合同模板甲方 = 机构；发票主体 = 机构。平台不得作为以上任一主体。
2. **资金不沉淀**：学员每期款项直接进入机构收单账户，平台系统不持有学员资金余额。平台收入仅为向机构开具的服务费。
3. **先交付后扣款**：任一期代扣指令只能在对应课程内容"已交付/已解锁"事件之后发起。系统须有前置校验，违反则拒发代扣。
4. **未成年拦截**：实名核验返回年龄 < 18 的用户，注册即拦截，不得下单。
5. **信用数据**：第三方信用分（如芝麻分）不入平台自有风控特征库、不用于平台放贷决策；芝麻先享以"场景产品 + 机构为商户"形态接入（详见 §9.6）。
6. **渠道反传销**：业务员提成与拉新奖励均为单级，仅与"本人直接促成的订单 / 直接邀请关系"挂钩，系统不支持多级分佣。
7. **消费者保护**：客单价/总分期金额上限、7 天无条件冷静期、信息充分披露、催收方式约束均为产品内置规则（详见 §11.4）。

## 3. 系统总体架构

### 3.1 分层架构

```
┌─────────────────────────────────────────────────────────────┐
│  接入层  学员端小程序 / H5 │ 机构端 Web │ 业务员端轻应用 │ 后台 Web │
├─────────────────────────────────────────────────────────────┤
│  网关层  API Gateway（鉴权 / 限流 / 路由 / 审计日志）             │
├─────────────────────────────────────────────────────────────┤
│  应用服务层                                                     │
│   用户服务 │ 机构服务 │ 课程服务 │ 订单服务 │ 分期/代扣服务         │
│   信用风控服务 │ 合同服务 │ 提成服务 │ 拉新服务 │ 学习/打卡服务      │
│   结算服务 │ 通知服务 │ 报表服务 │ 内容交付服务                    │
├─────────────────────────────────────────────────────────────┤
│  集成层（适配器）                                                │
│   实名核验 │ 电子合同 │ 支付代扣 │ 信用服务 │ 视频点播 │ 短信/模板消息 │
├─────────────────────────────────────────────────────────────┤
│  数据层  MySQL（主业务）│ Redis（缓存/分布式锁）│ 对象存储 │ 数据仓库 │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 技术栈

| 层 | 选型 | 备注 |
|---|---|---|
| 学员/业务员端 | 微信小程序（主）+ H5（备） | 小程序便于集成支付分/先享后付 |
| 机构端/后台 | Web（React/Vue 任一） | 管理类后台 |
| 后端 | Java Spring Boot 或 Node.js | 按团队技术栈 |
| 数据库 | MySQL 8 + Redis | 业务数据 + 缓存/锁/幂等 |
| 对象/视频 | 阿里云 OSS + 阿里云 VOD（或腾讯 VOD） | 视频加密、防盗链、播放器 SDK |
| 消息队列 | RocketMQ/Kafka | 代扣重试、异步通知、对账 |
| 电子合同 | 法大大 / e签宝 | 机构为甲方，人脸核身 |
| 实名 | 公安部四要素核验接口 | 注册强制，未成年硬拦 |
| 支付代扣 | 易宝 / 通联（初期）+ 微信支付分 / 芝麻先享（中期） | 收款方=机构，多通道并行 |

### 3.3 多端归属与会话

所有端共享统一账户体系；业务员/学员端基于微信小程序 openid + 手机号 + 实名绑定。归属关系（业务员-学员、邀请人-被邀请人）在注册时落库并永久锁定（见 §5、§8.1）。

## 4. 角色与权限模型

| 角色 | 入口 | 关键权限 |
|---|---|---|
| 学员 | 小程序/H5 | 浏览/下单/签约/学习/打卡/还款/查看自己的订单与发票/拉新 |
| 业务员 | 小程序轻应用 | 查看名下学员与履约、提成看板、拉新记录、服务性提醒；**不可见敏感信息、不可代行机构催收** |
| 机构管理员 | 机构端 Web | 课程管理、分期方案、与学员合同/发票、订单与代扣、逾期催收决策、答疑、结算对账 |
| 平台运营 | 后台 Web | 机构准入审核、服务费/上限配置、业务员审核、报表、风控规则、保证金、降级开关 |
| 平台财务 | 后台 Web | 服务费结算、对账、发票（平台→机构）管理 |
| 平台风控/合规 | 后台 Web | 风控规则、抽检、催收合规监控、数据导出审计 |

权限实现：RBAC，接口级 + 数据级（机构只能见本机构数据；业务员只能见名下学员脱敏数据）。

## 6. 状态机

**订单 order.status**
```
created ──(支付授权+签约+冷静期开始)──► cooling_off
cooling_off ──(冷静期内退课)──► refunded
cooling_off ──(冷静期满)──► active
active ──(全期履约完成)──► completed
active ──(机构同意退课/解约)──► terminated
```

**分期项 installment_item.status**
```
pending ──(内容交付)──► delivered ──(代扣成功)──► paid
delivered ──(代扣失败重试耗尽)──► overdue ──(机构核销)──► written_off
delivered ──(还款)──► paid
```

**代扣 deduction.status**
```
pending ──► success
pending ──(失败)──► failed ──(D+1/D+3/D+7 重试)──► success | overdue流程
```

**提成 commission.status**
```
pending ──(成单部分:T+1~3 / 履约部分:对应期还款成功)──► settled
pending ──(对应期逾期)──► held ──(还款)──► settled
settled(成单部分) ──(连续逾期>60天)──► clawed_back(按配置比例)
```

## 7. 四端功能设计

### 7.1 学员端（小程序为主）

- **注册/实名**：扫业务员二维码 → 手机号注册 → 公安部四要素实名 → 人脸核身 → **年龄<18 硬拦截**。归属业务员、邀请人在此落库并锁定。
- **首页/检索**：课程分类、热门、搜索。
- **课程详情**：机构介绍、大纲、师资、分期方案、评价；**显著标注"签约与付款对象为 XX 机构"**、总价/期数/每期金额/退课规则。
- **下单签约**：与机构直签电子合同（甲方=机构）+ 多通道支付授权（微信支付分/芝麻/银行卡）。
- **冷静期**：下单后 7 天无条件退课，全额退、不计违约。
- **学习中心**：按期解锁课时、VOD 播放、扫脸打卡获学习激励。
- **答疑**：提交问题/查看回复/服务记录。
- **我的订单**：分期进度、下期扣款时间、激励余额、机构信息、机构发票下载。
- **还款记录**：每期扣款详情（收款方=机构）、逾期提醒、重试入口。
- **拉新**：生成专属邀请链接/二维码；被邀请人完成首期还款后，邀请人自动获现金奖励（单级）。

### 7.2 机构端（Web 商户后台）— 卖方与债权方控制台

- **课程管理**：增删改课程，上传介绍/大纲/师资/视频源。
- **分期方案**：设定期数与每期金额（受平台上限约束）；合同模板甲方=机构。
- **合同与发票**：管理与学员的分期合同；向学员开具课程发票。
- **订单与代扣**：本机构全量订单、分期状态、退课申请；查看/发起代扣（收款方=机构）。
- **逾期与催收**：逾期清单与**催收决策由机构作出**；平台提供合规提醒工具。
- **答疑管理**：回复学员、系统留痕。
- **结算对账**：应付平台服务费、累计结算、对账单导出。
- **保证金账户**：履约保障金余额与扣减记录。

### 7.3 业务员端（小程序内嵌轻应用）— 平台获客渠道

- 专属二维码（绑定 staff_id + group_id，可后台禁用）。
- 我的学员（名下学员**脱敏**列表、订单状态、履约进度）。
- 提成看板（成单/履约、已结/待结、扣回记录）。
- 拉新跟进、学员服务性提醒（**强催收由机构执行，业务员不代行债权**）。
- 申请成为业务员（填写→客服审核→签协议→激活）。
- 归属规则：注册归属永久锁定；升级业务员后存量归属不变、仅新带客归己；自购不计提成；离职禁用二维码、归属保留不转移。

### 7.4 平台后台（运营中枢）

- 机构管理（入驻审核：营业执照/办学资质/收单资质；服务费率与阶梯；客单价上限配置）。
- 履约保障金管理（机构对持续交付的担保，非坏账准备金，不用于放款或兜底机构债务）。
- 代扣与逾期工具（作为 SaaS 提供给机构，扣款收款方=机构）。
- 业务员/提成配置、风控规则、降级开关（按机构灰度）。
- 数据报表与健康指标（§12.2）。
- 合规模块：催收合规监控、抽检、数据导出审计。

## 8. 核心业务流程与时序

### 8.1 注册与归属
```
扫码(staff二维码/邀请链接) → 手机号注册 → 四要素实名 → 人脸核身
  → 年龄校验(<18 拦截) → 落库: referrer_staff_id / referrer_student_id(永久锁定)
```

### 8.2 下单 → 信用评估 → 签约 → 分期
```
1. 选课下单 → 订单服务创建 order(seller_org_id=机构)
2. CreditService.resolveProvider(INSTALLMENT_COURSE)
     zhima.enabled? → 芝麻先享适配器(机构为商户) : 内置风控
3. (若芝麻) 用户授权 → evaluateScenario(eligible?) → signCreditAgreement(协议号)
4. 合同服务: 学员⇄机构直签(甲方=机构), 生成 contract
5. 支付授权(多通道) → 订单进入 cooling_off(7天)
6. 冷静期满 → active；生成 installment_item × N
```

### 8.3 逐期履约（先交付后扣款）
```
每期: 内容解锁交付 → installment_item.content_delivered_at 置位
      → 代扣服务校验"已交付"通过 → 发起代扣(payee=机构)
      → 优先级 微信支付分→芝麻→银行卡；失败 D+1/D+3/D+7 重试(每次前推送提醒)
      → 全部失败 → 进入机构逾期流程(机构决策催收, 业务员服务性提醒)
```

### 8.4 退课 / 冷静期
```
cooling_off 内: 无条件退课 → 全额退、不计违约 → order=refunded
active 后退课: 按机构退课规则与合同处理(机构决策) → order=terminated
```

### 8.5 打卡与学习激励
```
完成课时 → 扫脸比对(与实名人脸) → matched → 发放 incentive(source=机构)
下期 actual_amount = plan_amount − 已用 incentive
```

### 8.6 拉新奖励
```
被邀请人完成首期还款 → 触发 referral_reward
  → 代扣20%个税(偶然所得, 单笔规则见§税务) → net_amount 发放给邀请人(单级)
防刷: 被邀请人须实名+首期还款; 同一人名下多账号不重复计奖
```

### 8.7 提成计算与发放
```
成单部分: 订单成立 T+1~3 发放(占比可配)
履约部分: 每期还款成功 → 发放当期份额
逾期: 暂缓当期履约提成(held); 连续逾期>60天 → 按比例扣回成单部分
税务: 单次>500元 业务员代开票后打款; ≤500元 平台代扣20%个税
```

### 8.8 结算与发票（两段闭环）
```
机构→学员: 机构开全额课程发票给学员(机构为签约/收款方)
平台→机构: 平台按 GMV×服务费率 向机构开服务费发票
结算: 按周期生成 settlement, 机构向平台付服务费
```

## 9. 后端服务模块

### 9.1 用户服务
账户、实名、人脸核身、年龄拦截、归属关系、授权管理（个保法：最小必要、可撤回、留痕）。

### 9.2 机构服务
入驻审核、资质核验、服务费率/上限配置、保证金台账、退出（90 天通知）保障存量学员。

### 9.3 课程服务
课程 CRUD、上下线（下线不影响存量订单）、分期方案、VOD 资产绑定。

### 9.4 订单服务
下单、状态机、冷静期、退课、与合同/代扣/信用服务编排。强校验：seller=机构、未成年拦截、上限。

### 9.5 分期 / 代扣服务
分期项生成、**先交付后扣款校验**、多通道代扣编排、重试（D+1/D+3/D+7）、逾期流转。收款方恒为机构。

### 9.6 信用风控服务（含芝麻先享接入预留）

可插拔 Provider 适配层，对上游暴露统一契约，对下游对接不同信用源。

```typescript
type ScenarioCode = "INSTALLMENT_COURSE" | "DEPOSIT_FREE_RENT";

interface CreditProvider {
  isEnabled(): boolean;
  createAuthorization(ctx: AuthContext): Promise<AuthResult>;   // ctx.merchantId 解析为机构
  evaluateScenario(authToken: string, s: ScenarioCode): Promise<CreditDecision>;
  signCreditAgreement(authToken: string, plan: PlanContext): Promise<AgreementResult>;
}

interface CreditDecision {
  eligible: boolean;
  riskTier?: "A" | "B" | "C";
  suggestedLimit?: number;
  providerRef: string;
  // ★ 无 rawScore 字段：不接收/不入库原始芝麻分作自有风控特征
}
```

**Provider：**
- `BuiltInProvider`（默认常开）：仅用合法采集数据——实名结果、设备指纹、平台内行为、机构回传的履约/退课记录。输出 `CreditDecision`，不含任何第三方征信分。
- `ZhimaXianxiangProvider`（**预留，开关 `enabled=false`**）：桩形态落地，接口签名/配置/落库字段就位。接通后方法映射：
  - `signCreditAgreement` → `zhima.credit.payafteruse.creditagreement.sign`
  - 代扣 → 信用代扣/资金预授权接口（收款方=机构）
  - `evaluateScenario` → 场景信用结论（非原始分）
  - 原始分查询 `zhima.credit.score.get` **默认不接**（`store_raw_score=false` 恒置）
  - 商户解析 `merchant_resolver=BY_ORG`、`app_id_per_org=true`（商户主体=机构）
- 降级：芝麻未开通或异常 → 自动回退 `BuiltInProvider`，下单链路不阻断。

详见配套《信用风控服务开发文档》，本节为完整文档内的等价收录。

### 9.7 合同服务
法大大/e签宝集成，模板甲方=机构、人脸核身、签署回执留存。

### 9.8 提成服务 / 9.9 拉新服务
单级规则强约束（无多级分佣/多级奖励）；提成混合模式、逾期 held/扣回；拉新防刷与税务代扣。

### 9.10 学习 / 打卡服务
课时解锁、VOD 播放进度、扫脸比对、学习激励发放（source=机构）、服务痕迹留存。

### 9.11 内容交付服务
阿里云/腾讯 VOD：视频加密、防盗链、防截屏水印、播放器 SDK、解锁日志（与代扣一一对应）。

### 9.12 通知服务 / 9.13 结算服务 / 9.14 报表服务
模板消息+短信（还款提醒、服务触达）；服务费结算与对账；GMV/回收率/逾期率等指标。

## 11. 安全与个人信息保护

### 11.1 数据安全
身份证号/银行卡号加密存储（原文加密 + 哈希索引）；业务员端等只读脱敏；密钥管理用 KMS；传输 TLS。

### 11.2 个保法合规
最小必要采集；授权前置、可撤回；保留期限可配；数据导出/删除审计；信用授权与人脸数据单独同意与留痕。

### 11.3 接口安全
网关鉴权/限流/防重放；幂等键防重复扣款；RBAC 接口级+数据级隔离；操作审计日志。

### 11.4 消费者保护（产品内置规则）
- **客单价/总分期金额上限**：后台可配，超限不可下单。
- **7 天无条件冷静期**：期内退课全额退、不计违约。
- **信息充分披露**：课程页/合同明示总价、期数、每期金额、退课规则、签约对象为机构。
- **催收方式约束**：仅限合规时段/话术的提醒；禁止骚扰、威胁、向无关第三人催收；系统对提醒频次/时段做硬限制并留痕。
- **机构准入与抽检**：资质核验 + 定期抽检课程质量与履约，劣质机构下线。

## 12. 非功能性需求与运营健康指标

### 12.1 非功能
- 可用性：核心链路（下单/代扣）99.9%；代扣失败可重试、可对账、可幂等。
- 性能：详情页 P95 < 500ms；代扣批处理可水平扩展。
- 可观测：全链路日志、指标、告警；代扣/降级/逾期专项监控。
- 一致性：金额相关操作走事务 + 幂等；分布式锁防并发扣款。

### 12.2 运营健康指标（带阈值与动作）

| 指标 | 健康阈值 | 异常动作 |
|---|---|---|
| 分期回收率（核心） | > 90% | 排查机构准入与用户准入 |
| 代扣首次成功率 | > 80% | 优化多通道授权引导 |
| 逾期率 | < 10% | 触发提成调整与机构跟进 |
| 退课率 | < 5% | 排查课程质量与冷静期机制 |
| 答疑响应时长 | < 24h | 自动预警、转派老师 |
| 拉新转化率 | > 20% | 优化邀请卡片与奖励力度 |
