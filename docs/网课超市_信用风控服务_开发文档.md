# 先学后付·网课超市 — 信用风控服务开发文档

> 适用范围：分期下单链路中的信用评估、芝麻先享接入预留、代扣授权与降级策略
> 结构定性（贯穿全文）：**机构为卖方/签约方/收款方并自担风险，平台为撮合·技术·获客中介。** 芝麻信用相关能力的商户主体是**机构**，平台是技术对接方（ISV）。

---

## 0. 设计原则与合规约束（开发必读）

本模块的设计目标，是让信用能力"可插拔、可降级、可审计"，同时**不让平台落入需持牌的征信/放贷角色**。以下为硬约束，任何实现不得违背：

1. **平台不接收、不入库、不复用原始芝麻分作为自有放贷风控特征。** 断直连要求下，第三方信用数据不得被直接用于放贷决策。平台自有风控只能使用合法采集的数据（实名结果、设备、平台内行为、机构回传的履约记录）。
2. **芝麻先享走"场景产品"形态，不走"裸数据"形态。** 接入后调用返回的是场景内的信用结论（是否可先享/可分期、建议额度、信用代扣协议），而非可二次加工的原始分。
3. **商户主体（appId/支付宝账户）= 课程机构。** 企业支付宝账户名称需与机构营业执照一致。平台用自身账户去签约该能力即越位为信贷当事人，禁止。
4. **用户授权前置、最小必要、全程留痕。** 任何信用查询/签约必须有用户在支付宝侧的显式授权，留存授权凭证与时间戳。
5. **强降级。** 芝麻先享未开通或调用失败时，下单链路必须能回退到"内置风控 + 其他通道（如免押/担保/三方代扣）"，不得硬依赖芝麻。

> 注：本文档是工程设计，不替代法律意见。资质、合同与资金路径上线前须经执业律师与支付服务商确认。

## 1. 模块在系统中的位置

```
学员端小程序 ──┐
              │  下单/签约/学习
机构端(Web) ───┼──► 订单服务 ──► 信用风控服务(本模块) ──► Provider 适配层
              │                         │                    ├─ 内置风控(默认开启)
业务员端 ──────┘                         │                    ├─ 芝麻先享适配器(预留, 开关控制)
                                        │                    ├─ 微信支付分适配器(可选, 预留)
                                        ▼                    └─ 实名/四要素适配器
                                   合同服务(法大大/e签宝)
                                   代扣服务(易宝/通联/芝麻代扣)
```

信用风控服务对上游（订单服务）只暴露**统一接口**；对下游通过**Provider 适配层**对接不同信用源。新增/启用一个信用源 = 实现一个适配器 + 打开开关，上层无需改动。

## 2. 统一接口定义（上游调用的稳定契约）

> 语言无关，下面以 TypeScript 风格描述契约。返回值刻意做成"场景结论"，与具体 Provider 解耦。

```typescript
type ScenarioCode = "INSTALLMENT_COURSE" | "DEPOSIT_FREE_RENT"; // 课程分期 / 免押租机

interface CreditProvider {
  /** 该信用源当前是否可用（签约/白名单/开关状态） */
  isEnabled(): boolean;

  /** 发起用户授权：返回授权跳转链接或授权参数，由前端拉起支付宝授权页 */
  createAuthorization(ctx: AuthContext): Promise<AuthResult>;

  /** 授权回调后，查询"场景信用结论"（注意：不是原始分） */
  evaluateScenario(authToken: string, scenario: ScenarioCode): Promise<CreditDecision>;

  /** 签署信用代扣/先享后付协议（芝麻先享的核心动作），返回协议号供代扣使用 */
  signCreditAgreement(authToken: string, plan: PlanContext): Promise<AgreementResult>;
}

interface AuthContext {
  merchantId: string;     // ★ 解析为「机构」的支付宝商户号，不是平台
  orderId: string;
  scenario: ScenarioCode;
  callbackUrl: string;
}

interface CreditDecision {
  eligible: boolean;          // 场景内是否可先享/可分期
  riskTier?: "A" | "B" | "C"; // Provider 给的粗粒度分层（若提供）
  suggestedLimit?: number;    // 建议额度/可分期上限（若提供）
  providerRef: string;        // Provider 侧的查询/决策流水号（用于留痕对账）
  // ★ 不设 rawScore 字段：默认不接收原始芝麻分；
  //    如个别场景确需展示分值，单独走 display-only 通道，禁止进风控特征库
}

interface AgreementResult {
  agreementNo: string;        // 信用代扣/先享后付协议号
  status: "SIGNED" | "PENDING" | "FAILED";
  providerRef: string;
}
```

上游（订单服务）只认 `CreditProvider`。它通过 `CreditService` 选择当前生效的 Provider：

```typescript
class CreditService {
  // 按场景 + 开关 + 优先级选择 Provider；芝麻未开通时自动回退内置风控
  resolveProvider(scenario: ScenarioCode): CreditProvider { ... }
}
```

## 3. 芝麻先享适配器（预留实现）

> 现在以**桩(stub)**形态落地：接口签名、配置项、数据落库字段全部就位，内部方法在未开通时返回 `isEnabled()=false` 并抛 `ProviderNotEnabled`。白名单批下来后，把桩替换为对芝麻开放平台 API 的真实调用即可，**上层与数据库结构不变**。

### 3.1 配置项（运营/机构后台可配）

```yaml
zhima_xianxiang:
  enabled: false                 # ★ 白名单开通后置 true
  merchant_resolver: "BY_ORG"    # 商户号解析方式：按机构解析（不可用平台主体）
  app_id_per_org: true           # 每个机构独立 appId / 支付宝商户号
  default_scenario: "INSTALLMENT_COURSE"
  store_raw_score: false         # ★ 恒为 false：不入库原始分作风控特征
```

### 3.2 方法 ↔ 芝麻开放平台 API 映射（接通时填充）

| 适配器方法 | 对应芝麻/支付宝接口（接通后实现） | 说明 |
|---|---|---|
| `createAuthorization` | 用户信用授权（页面跳转类，拉起支付宝授权页） | 须用户显式授权，回调带 authToken |
| `signCreditAgreement` | `zhima.credit.payafteruse.creditagreement.sign`（信用服务开通/先用后付签约） | 返回信用代扣协议号 |
| 代扣发起（在代扣服务中） | 信用代扣 / 资金预授权相关接口 | 收款方为机构，每期内容交付后发起 |

| （默认不接）原始分查询 | `zhima.credit.score.get` | 仅在机构具备资质且场景明确允许时单独评估；默认关闭，不入风控库 |

## 4. 内置风控 Provider（默认始终可用）

芝麻未接通期间的主力，也是接通后的基线（与芝麻结论可并用）。

- **输入特征（合法来源）**：实名核验结果、设备指纹、注册-下单行为、历史还款表现、机构回传的履约/退课记录。
- **输出**：同样实现 `CreditProvider` 契约，返回 `CreditDecision`（eligible / riskTier / suggestedLimit）。
- **不包含**：任何第三方征信/信用分数据。

---

## 5. 数据模型与存储规则

```
credit_authorization        授权记录: order_id, provider, merchant_id(机构), auth_token, status, ts
credit_decision             决策记录: order_id, provider, eligible, risk_tier, suggested_limit, provider_ref, ts
credit_agreement            协议记录: order_id, provider, agreement_no, status, provider_ref, ts
```

**存储红线：**
- ✅ 存：授权状态、场景决策结论、分层、协议号、Provider 流水号（对账/审计用）。
- ❌ 不存：原始芝麻分作为自有风控特征（`store_raw_score=false` 强制）。
- ✅ 个保法：授权链路留痕、最小必要、保留期限可配、支持用户撤回授权。

## 6. 调用时序（课程分期下单链路）

```
1. 学员下单选课
   └─ 订单服务创建订单(卖方=机构)

2. 信用评估
   └─ CreditService.resolveProvider(INSTALLMENT_COURSE)
        ├─ zhima.enabled? → ZhimaXianxiangProvider
        └─ else           → BuiltInProvider(内置风控)

3. (若芝麻) 用户授权
   └─ createAuthorization → 前端拉起支付宝授权页 → 回调 authToken

4. 场景结论 + 签约
   ├─ evaluateScenario → eligible?
   └─ signCreditAgreement → agreementNo (信用代扣协议)

5. 分期合同
   └─ 合同服务: 学员 ⇄ 机构 直接签署(法大大/e签宝), 抬头/开票方=机构

6. 分期代扣(逐期)
   └─ 每期内容交付解锁后 → 代扣服务发起(收款方=机构)
        优先级: 信用代扣/支付分 → 芝麻 → 银行卡四要素
        失败重试 D+1/D+3/D+7 → 仍失败转机构逾期流程
```

关键顺序约束：**先交付内容、后发起当期代扣**；签约方与收款方始终是机构。

## 7. 开关与降级

- `zhima_xianxiang.enabled=false` 时：`resolveProvider` 跳过芝麻，走内置风控；下单链路完整可用（叠加免押/担保/三方代扣等通道）。
- 芝麻接通后运行期异常（超时/限流/拒绝）：单笔降级到内置风控，不阻断下单；记录降级原因供监控。
- Feature flag 支持按机构灰度开通，不必全量。

## 8. 待芝麻先享接入时的对接清单（Checklist）

1. **资质**：机构具备企业支付宝账户，账户主体与营业执照一致；满足芝麻先享白名单/营收/品牌门槛。
2. **签约**：在芝麻信用商家服务平台注册、签约先享后付/信用服务能力，按机构主体创建应用、交换公私钥。
3. **配置**：每机构独立 `appId`/商户号写入 `app_id_per_org`；`enabled` 置 true。
4. **实现**：把第 3 节桩内 `TODO(接通)` 替换为真实 API 调用；映射表逐项填充。
5. **联调**：授权回调、签约、信用代扣、对账流水（`provider_ref`）全链路打通。
6. **复核**：原始分仍不入风控库（`store_raw_score=false` 守住）；商户主体为机构；用户授权留痕完整。
7. **合规**：合同、资金路径、收单资质由律师与支付服务商终审通过后再上量。
